# Agent Spec

## Overview
The `personal-assistant` agent is a personal assistant that tracks tasks, habits, and workouts for one user.

### Main Capabilities
1. **Natural-language capture**: Quick, unambiguous capture of new tasks, habits (creation), habit logs (completions), and workout logs.
2. **Goal-style plan generation**: Decomposes vague requests (e.g. "clean my apartment by room," "structure my push/pull/legs week") into structured, recurring tasks. If a request is too vague, the agent asks clarifying questions first instead of guessing.
3. **Read-only advisory/analysis queries**: Answers questions on workout progress trends, exercise-switch advice, and warm-up set calculations.
4. **Dashboard view**: Deterministically aggregates everything due today â€” open/overdue tasks, unlogged habits, and today's calendar events â€” no LLM call involved.
5. **Current lifts tracking**: Shows first-vs-current weight progress for exercises currently in the active training plan, on the Workouts page.

## Example Use Cases
* **Natural-language capture**:
  - *Input*: "Log workout: Bench press 3x5 at 80kg"
  - *Expected Output*: "Logged workout: Bench press, 3 sets of 5 reps at 80kg."
* **Goal-style plan generation**:
  - *Input*: "Help me clean my apartment by room"
  - *Expected Output*: "To structure this plan, how many rooms do you have, and which ones would you like to prioritize first?"
* **Advisory/analysis queries**:
  - *Input*: "Calculate warm-up sets for a 100kg squat"
  - *Expected Output*: A calculated list of warm-up sets based on standard percentages (e.g., 40kg, 60kg, 80kg) and guidance grounded in user data.
* **Daily plan suggestions**:
  - *Input*: "Show me my plan for today"
  - *Expected Output*: An ordered layout of the user's due habits, current open tasks, and scheduled calendar blocks.

## Tools Required
* **SQLite Database**: Local persistent database for storing tasks, habits, and workout logs.
* **Mock Calendar Dataset**: Seeded dataset representing the user's calendar.

## Constraints & Safety Rules
1. **Explicit Deletion Confirmation**: Deleting any tasks, habits, or workout logs must require explicit user confirmation. Creation can happen freely.
2. **Medical Disclaimer**: Fitness/workout advice must remain general and must not replace professional medical or training advice. If injury or a health condition is mentioned, the agent must advise consulting a professional.
3. **Strict Grounding**: Advisory and analysis responses must be strictly grounded in the user's actually logged SQLite data. No fabricated trends or claims.
4. **Clarification for Vague Plans**: The agent must ask clarifying questions when a goal is too vague to act on, rather than guessing.

## Success Criteria
* Accurate natural-language parsing and storage of logs/tasks in the SQLite database.
* Proper generation of structured task lists from clear goals.
* Prompting for clarification when plans or goals are vague.
* Enforcement of safety guardrails: deletion warnings, medical disclaimers, and data grounding.
* A daily schedule generated correctly combining tasks, habits, and mock calendar events.

## Data Model

All persisted in SQLite.

- **Task**: `id`, `title`, `tag` (freeform string, e.g. "chore", "errand"), `recurrence` (`null`|`daily`|`weekly`|`biweekly`|`monthly`), `due_date`, `completed` (bool), `completed_at`, `source` (`manual`|`nl_capture`|`plan_generated`)
- **Habit**: `id`, `name`, `recurrence`, `streak_count`, `last_completed_date`, `log` (list of completion dates)
- **CalendarEvent** (mock, manually seeded): `id`, `title`, `start`, `end`
- **WorkoutLog**: `id`, `date`, `exercise_name`, `sets` (list of `{reps, weight_kg}`), `notes`
- **PlanExercise**: `id`, `exercise_name` (TEXT, exact-match against WorkoutLog.exercise), `active` (bool, default true), `added_at`


## Architecture

A single freeform prompt is the main interaction surface. A **request classification** step routes every submission to one of three paths:
1. **Capture** â€” direct, unambiguous entries parsed straight into a Task/Habit/Habit-log/WorkoutLog row, created immediately. Habit creation (defining a new habit to track) and habit logging (recording a completion against an existing habit) are distinct operations the agent must distinguish.
2. **Plan generation** â€” vague goal-style requests. A completeness check asks clarifying questions if too vague; once there's enough information, drafts and **creates directly** (no separate approval step), returning a summary of what was created.
3. **Advisory/analysis query** â€” read-only, never writes data.

The **Dashboard** tab aggregates today's due items via direct SQL queries only, avoiding repeated model calls on every tab visit. A user can still ask for a narrated summary of their day through the chat prompt box (advisory path), on demand.

**Plan exercise tracking**: exercises become "active" when the **workout-planning** skill's plan-generation path creates or restructures a weekly split (active set for everything in the new plan, deactivated â€” not deleted â€” for anything dropped from it), or via a standalone toggle tool for single-exercise changes outside a full plan regeneration. First/current weight for the Workouts page is computed by finding the earliest and latest WorkoutLog entry per active exercise and taking the max set weight from each as that session's representative weight. Matching is exact-string on exercise name â€” no fuzzy matching in v1.

## Skills

- **`household-planning`**: domain knowledge for decomposing chore-type goals into a sensible room/task breakdown with reasonable cleaning cadences. Loaded by the plan-generation path for chore/household-related requests.
- **`workout-planning`**: exercise rotation/progression principles, weekly split structuring (push/pull/legs, upper/lower, full-body), reading WorkoutLog history for trends, marking exercises active/inactive in **plan_exercises** when generating or restructuring a weekly split, and when to invoke the **warm-up calculator** tool (percentage-based ramp-up, e.g. 40/60/80% of target working weight â€” a pure deterministic function, not LLM math). Loaded by whichever path is active whenever the request is workout-related.

## Frontend

FastAPI + Jinja2 templates + HTMX, styled with custom CSS (no CDN framework â€” design language evolved into a custom, hand-tuned system rather than a utility framework). Single process (**uvicorn**), no Node/npm toolchain. Design system documented separately in **DESIGN.md** (managed via the Impeccable skill) â€” **DESIGN.md** is the canonical reference for colors, typography, and component styling; this section covers structure and behavior only.

**Navigation**: a persistent top nav bar (Dashboard, Tasks, Habits, Workouts, Calendar) plus a persistent chat sidebar (always visible, never re-rendered on tab switch, retains chat history) wrapping a swappable content area, navigated via HTMX (**hx-get** + **hx-target** + **hx-push-url**) â€” not full page reloads. Active tab highlighting derives from the current URL via **htmx:afterOnLoad** and **popstate** listeners.

**Dashboard tab** (default landing page): deterministic aggregation only, no LLM call â€” Today's Schedule, Open & Overdue Tasks, and Habits To Do Today, each as its own card querying SQLite directly.

**Tasks tab**: full task list with complete/delete actions (custom confirmation modal, not native browser confirm) and the task's **source** badge â€” shown only for **nl_capture**/**plan_generated**, hidden when **source = "manual"**.

**Habits tab**: habit list with streak counters and a per-habit completion heatmap (last 12 weeks, GitHub-style grid).

**Workouts tab**: chronological workout log list, plus a "Current Lifts" card showing first-vs-current weight for active-plan exercises only (see Architecture â†’ Plan exercise tracking).

**Calendar tab**: full chronological listing of all mock calendar events, distinct from the Dashboard's "today only" schedule card.

**Date display**: all displayed dates use German formatting (**DD.MM.YYYY** for list items, full German weekday/month names for the header, e.g. "Dienstag, 23. Juni 2026") via one shared formatting helper. Underlying SQLite storage remains ISO (**YYYY-MM-DD**) â€” only display formatting changed.

**Design language**: flat opaque surfaces only â€” no backdrop-filter/blur, no gradients, no colored/tinted glow shadows. A single accent color (muted antique gold) used sparingly (under 10% of any screen) against a dark green base; red reserved exclusively for destructive actions. No decorative emoji icons in date/time row displays (Calendar, Dashboard schedule) â€” plain text in fixed-width aligned columns instead; other icons (habit streak flame, overdue warning) are unaffected.

## Out of Scope for v1

Real Google Calendar integration (MCP server + OAuth), proactive ambient nudges, smart rescheduling, and goal linking are explicit stretch goals, not part of this build.

Exercise-name matching for plan tracking and progress calculation is exact-string only â€” inconsistent naming when logging (e.g. "Deadlift" vs "deadlifts") will be treated as different exercises.
