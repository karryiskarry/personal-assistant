# Agent Spec

## Overview
The `personal-assistant` agent is a personal assistant that tracks tasks, habits, and workouts for one user.

### Main Capabilities
1. **Natural-language capture**: Quick, unambiguous capture of new tasks, habits (creation), habit logs (completions), and workout logs.
2. **Goal-style plan generation**: Decomposes vague requests (e.g. "clean my apartment by room," "structure my push/pull/legs week") into structured, recurring tasks. If a request is too vague, the agent asks clarifying questions first instead of guessing.
3. **Read-only advisory/analysis queries**: Answers questions on workout progress trends, exercise-switch advice, and warm-up set calculations.
4. **Dashboard view**: Deterministically aggregates everything due today — open/overdue tasks, unlogged habits, and today's calendar events — no LLM call involved.
5. **Current lifts tracking**: Shows first-vs-current weight progress for exercises currently in the active training plan, on the Workouts page.

## Example Use Cases
* **Natural-language capture**:
  - *Input*: "Log workout: Bench press 3x5 at 80kg"
  - *Expected Output*: "Logged workout: Bench press, 3 sets of 5 reps at 80kg."
* **Goal-style plan generation**:
  - *Input*: "Help me clean my apartment by room"
  - *Expected Output*: "To structure this plan, how many rooms do you have, and which ones would you like to prioritize first?"
* **Advisory/analysis queries**:
  - *Input*: "Calculate warm-up sets for a 100kg squat"
  - *Expected Output*: A calculated list of warm-up sets based on standard percentages (e.g., 40kg, 60kg, 80kg) and guidance grounded in user data.
* **Daily plan suggestions**:
  - *Input*: "Show me my plan for today"
  - *Expected Output*: An ordered layout of the user's due habits, current open tasks, and scheduled calendar blocks.

## Tools Required
* **SQLite Database**: Local persistent database for storing tasks, habits, and workout logs.
* **Mock Calendar Dataset**: Seeded dataset representing the user's calendar.

## Constraints & Safety Rules
1. **Explicit Deletion Confirmation**: Deleting any tasks, habits, or workout logs must require explicit user confirmation. Creation can happen freely.
2. **Medical Disclaimer**: Fitness/workout advice must remain general and must not replace professional medical or training advice. If injury or a health condition is mentioned, the agent must advise consulting a professional.
3. **Strict Grounding**: Advisory and analysis responses must be strictly grounded in the user's actually logged SQLite data. No fabricated trends or claims.
4. **Clarification for Vague Plans**: The agent must ask clarifying questions when a goal is too vague to act on, rather than guessing.
5. **No Raw IDs Surfaced to the User**: The agent resolves a name/description to its internal database ID itself (via the read-only query tool) before any mutation or confirmation. It must never ask the user to supply or look up a raw ID, and must never display one in a confirmation message — only the item's name/description.

## Success Criteria
* Accurate natural-language parsing and storage of logs/tasks in the SQLite database.
* Proper generation of structured task lists from clear goals.
* Prompting for clarification when plans or goals are vague.
* Enforcement of safety guardrails: deletion warnings, medical disclaimers, and data grounding.
* A daily schedule generated correctly combining tasks, habits, and mock calendar events.

## Data Model

All persisted in SQLite.

- **Task**: `id`, `title`, `tag` (freeform string, e.g. "chore", "errand"), `recurrence` (`null`|`daily`|`weekly`|`biweekly`|`monthly`), `due_date`, `completed` (bool), `completed_at`, `source` (`manual`|`nl_capture`|`plan_generated`)
- **Habit**: `id`, `name`, `description`, `frequency` (`daily`|`weekly`|`biweekly`|`monthly`), `created_at`. Completions are tracked in a separate **HabitLog** table (`id`, `habit_id`, `date`, `status`, `created_at`) — streak counts and the heatmap are computed on read from `HabitLog`, not stored as fields on `Habit`.
- **CalendarEvent** (mock, manually seeded): `id`, `title`, `start`, `end`
- **WorkoutLog**: `id`, `date`, `exercise_name`, `sets` (list of `{reps, weight_kg}`), `notes`
- **PlanExercise**: `id`, `exercise_name` (TEXT, exact-match against WorkoutLog.exercise), `active` (bool, default true), `added_at`


## Architecture

A single freeform prompt is the main interaction surface. A **request classification** step routes every submission to one of three paths:
1. **Capture** — direct, unambiguous entries parsed straight into a Task/Habit/Habit-log/WorkoutLog row, created immediately. Habit creation (defining a new habit to track) and habit logging (recording a completion against an existing habit) are distinct operations the agent must distinguish.
2. **Plan generation** — vague goal-style requests. A completeness check asks clarifying questions if too vague; once there's enough information, drafts and **creates directly** (no separate approval step), returning a summary of what was created.
3. **Advisory/analysis query** — read-only, never writes data.

The **Dashboard** tab aggregates today's due items via direct SQL queries only, avoiding repeated model calls on every tab visit. A user can still ask for a narrated summary of their day through the chat prompt box (advisory path), on demand.

**Plan exercise tracking**: exercises become "active" when the **workout-planning** skill's plan-generation path creates or restructures a weekly split (active set for everything in the new plan, deactivated — not deleted — for anything dropped from it), or via a standalone toggle tool for single-exercise changes outside a full plan regeneration. First/current weight for the Workouts page is computed by finding the earliest and latest WorkoutLog entry per active exercise and taking the max set weight from each as that session's representative weight. Matching is exact-string on exercise name — no fuzzy matching in v1.

**Habit streaks & heatmap**: a single `get_period_start(date, frequency, anchor)` helper buckets a date into the right period for a given frequency — calendar day (daily), Monday-start ISO week (weekly), a 14-day period counted forward from the habit's `created_at` (biweekly, since there's no universal calendar anchor for "every 2 weeks"), or calendar month (monthly). Both the streak calculation and the heatmap grid call this same helper rather than each reimplementing bucketing — the two drifting out of sync against each other was a real bug, not a hypothetical one. The heatmap's window also varies by frequency: daily/weekly use 84 days (12 cells), biweekly uses 6 cells (84 days ÷ 14), monthly uses 12 cells over a 12-month window. A logged completion always takes precedence over a habit's `created_at` when deciding whether a cell is "missed" vs. "n/a" — `created_at` is not reliable enough to trust over real log data.

## Skills

- **`household-planning`**: domain knowledge for decomposing chore-type goals into a sensible room/task breakdown with reasonable cleaning cadences. Loaded by the plan-generation path for chore/household-related requests.
- **`workout-planning`**: exercise rotation/progression principles, weekly split structuring (push/pull/legs, upper/lower, full-body), reading WorkoutLog history for trends, marking exercises active/inactive in **plan_exercises** when generating or restructuring a weekly split, and when to invoke the **warm-up calculator** tool (percentage-based ramp-up, e.g. 40/60/80% of target working weight — a pure deterministic function, not LLM math). Loaded by whichever path is active whenever the request is workout-related.

## Frontend

FastAPI + Jinja2 templates + HTMX, styled with custom CSS (no CDN framework — design language evolved into a custom, hand-tuned system rather than a utility framework). Single process (**uvicorn**), no Node/npm toolchain. Design system documented separately in **DESIGN.md** (managed via the Impeccable skill) — **DESIGN.md** is the canonical reference for colors, typography, and component styling; this section covers structure and behavior only.

**Navigation**: a persistent top nav bar (Dashboard, Tasks, Habits, Workouts, Calendar) plus a persistent chat sidebar (always visible, never re-rendered on tab switch, retains chat history) wrapping a swappable content area, navigated via HTMX (**hx-get** + **hx-target** + **hx-push-url**) — not full page reloads. Active tab highlighting derives from the current URL via **htmx:afterOnLoad** and **popstate** listeners.

**Dashboard tab** (default landing page): deterministic aggregation only, no LLM call — Today's Schedule, Open & Overdue Tasks, and Habits To Do Today, each as its own card querying SQLite directly.

**Tasks tab**: full task list with complete/delete actions (custom confirmation modal, not native browser confirm) and the task's **source** badge — shown only for **nl_capture**/**plan_generated**, hidden when **source = "manual"**.

**Habits tab**: habit list with frequency-aware streak counters ("X day/week/biweekly/month streak", per the habit's own frequency) and a per-habit completion heatmap whose grid shape and time window depend on frequency (see Architecture → Habit streaks & heatmap).

**Workouts tab**: workout history grouped into per-date session cards — multiple exercises logged on the same day appear nested under one shared date header instead of as separate flat rows — plus a "Current Lifts" card showing first-vs-current weight for active-plan exercises only (see Architecture → Plan exercise tracking).

**Calendar tab**: full chronological listing of all mock calendar events, distinct from the Dashboard's "today only" schedule card.

**Date display**: all displayed dates use German formatting (**DD.MM.YYYY** for list items, full German weekday/month names for the header, e.g. "Dienstag, 23. Juni 2026") via one shared formatting helper. Underlying SQLite storage remains ISO (**YYYY-MM-DD**) — only display formatting changed.

**Design language**: flat opaque surfaces only — no backdrop-filter/blur, no gradients, no colored/tinted glow shadows. A single accent color (muted antique gold) used sparingly (under 10% of any screen) against a dark green base; red reserved exclusively for destructive actions. No decorative icons anywhere except three: the per-habit streak flame, the overdue-task warning (shown only on genuinely overdue rows — `due_date` strictly before today, not all open/due-today tasks), and the delete-confirmation modal's warning icon. Section headers (e.g. "Today's Schedule", "Habits To Do Today"), the chat panel header, and date/time rows (Calendar, Dashboard schedule) are plain text only — date/time rows use fixed-width aligned columns (`.calendar-date`/`.calendar-duration`) rather than natural text width, so values line up into clean columns across rows regardless of content length.

## Testing

Tests must never run against the live SQLite file. `tests/conftest.py` provides an autouse, function-scoped fixture that patches `DB_PATH` to an isolated temporary file and runs `init_db()` against it before each test. This exists because of a real incident, not a hypothetical risk: before this fixture existed, a unit test for `sync_active_exercises` called the real function with test-only exercise names directly against the live database, silently deactivating every real active exercise as a side effect of routine `pytest` runs — twice.

## Out of Scope for v1

Real Google Calendar integration (MCP server + OAuth), proactive ambient nudges, smart rescheduling, and goal linking are explicit stretch goals, not part of this build.

Exercise-name matching for plan tracking and progress calculation is exact-string only — inconsistent naming when logging (e.g. "Deadlift" vs "deadlifts") will be treated as different exercises.
